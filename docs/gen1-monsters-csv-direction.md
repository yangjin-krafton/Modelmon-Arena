# 1기 몬스트 CSV 전환 방향성

## 배경

현재 프로젝트에서 몬스터 관련 정보는 주로 [`README.md`](/mnt/d/Weeks/Modelmon-Arena/README.md)에 있는 기획성 표에 머물러 있고, 실제 런타임에서 읽는 구조화 데이터는 아직 없습니다.

이 상태에서는 다음 문제가 생깁니다.

- 몬스터 수가 늘어날수록 수정 지점이 문서와 코드 사이에 흩어진다.
- 밸런스 조정 시 어떤 값이 최신 기준인지 불명확해진다.
- CSV 기반 일괄 수정, 스프레드시트 검토, AI 보조 생성 같은 작업이 어렵다.

그래서 `1기 몬스트`부터는 테이블을 아예 CSV로 분리해, 사람이 읽기 쉬우면서도 이후 코드에서 바로 소비할 수 있는 데이터 소스로 만드는 방향을 검토한다.

## 목표

- `1기 몬스트`의 핵심 스펙을 문서 표가 아니라 데이터 파일로 관리한다.
- 초기에는 사람이 직접 열어보고 수정하기 쉬운 포맷을 우선한다.
- 나중에 JavaScript 런타임 데이터, 밸런싱 스크립트, 배치 생성 파이프라인으로 확장 가능한 구조를 잡는다.

## 왜 CSV인가

CSV는 현재 단계에서 다음 장점이 있다.

- 스프레드시트와 왕복이 쉽다.
- AI에게 초안 생성을 맡기거나, 사람이 수동 검수하기 편하다.
- JSON보다 행 단위 비교가 쉬워서 초반 밸런싱에 유리하다.
- 기획자 관점에서 "한 줄 = 한 개체" 감각이 명확하다.

반면 한계도 있다.

- 배열, 다중 타입, 다중 스킬처럼 중첩 구조 표현이 약하다.
- 컬럼 의미가 엄격하지 않으면 빠르게 난잡해진다.
- 참조 무결성 검증은 별도 스크립트가 필요하다.

즉, CSV는 `최초 편집 포맷`으로 적합하지만, 장기적으로는 런타임 직전 JSON 변환 또는 로더 계층이 함께 있어야 한다.

## 핵심 결정

### 1. 처음부터 "거대한 한 장짜리 CSV"로 가지 않는다

`1기 몬스트 테이블`을 CSV로 만든다고 해도, 모든 정보를 한 파일에 욱여넣는 방식은 피한다.

권장 분리는 아래와 같다.

- `gen1_monsters.csv`
  - 몬스터 기본 정보
- `gen1_monster_types.csv`
  - 몬스터와 타입의 N:M 관계
- `gen1_monster_skills.csv`
  - 몬스터와 스킬 슬롯 관계
- 필요 시 `skills.csv`, `types.csv`, `status_effects.csv`
  - 공용 사전 데이터

이렇게 나누면 몬스터 본체 데이터와 공용 시스템 데이터를 분리할 수 있고, 이후 런타임 로딩 구조로 옮기기 쉽다.

### 2. README 표는 "소개용", CSV는 "소스 오브 트루스"로 둔다

앞으로는 README 표를 직접 편집해 게임 데이터를 관리하지 않는다.

- README: 세계관 소개, 예시 라인업, 샘플 표
- CSV: 실제 수치와 식별자, 관계 데이터

즉, 데이터의 기준점은 문서 표가 아니라 CSV가 되어야 한다.

### 3. 1기 범위는 "전투에 바로 필요한 최소 정보"까지만 담는다

초기 CSV에는 설정집 수준의 장문 텍스트보다, 전투 구현과 카드/툴팁 출력에 필요한 필드를 먼저 담는 것이 맞다.

## 1차 스코프

`1기 몬스트 CSV`의 1차 범위는 아래 정도를 권장한다.

### 필수

- 고유 ID
- 표시 이름
- 모티프
- 세대 구분
- 기본 역할
- 희귀도 또는 티어
- HP / ATK / DEF / SPD 같은 기본 전투 스탯
- 설명용 짧은 한 줄

### 관계 데이터

- 타입 1, 타입 2 또는 별도 매핑 테이블
- 기본 스킬 3개
- 패시브 1개

### 선택

- 진화 단계
- 아트 에셋 파일명
- 모델 파일 경로
- 출시 여부
- 밸런스 메모

## 권장 CSV 구조

### `gen1_monsters.csv`

예시 컬럼:

`monster_id,name_ko,name_en,motif,generation,role,rarity,hp,atk,def,spd,short_desc,asset_key,enabled`

예시 행:

`gpt_sprout,GPT 스프라우트,GPT Sprout,GPT line,gen1,balanced,starter,72,68,70,65,범용성과 문맥 유지에 강한 스타터,gpt_sprout,true`

### `gen1_monster_types.csv`

예시 컬럼:

`monster_id,type_id,slot`

예시 행:

`gpt_sprout,general,1`
`gpt_sprout,context,2`

### `gen1_monster_skills.csv`

예시 컬럼:

`monster_id,skill_id,slot,learn_source`

예시 행:

`gpt_sprout,token_burst,1,base`
`gpt_sprout,context_bind,2,base`
`gpt_sprout,checkpoint_restore,3,base`
`gpt_sprout,chain_focus,passive,base`

## 컬럼 설계 원칙

- 사람 친화적인 표시명과 시스템용 ID를 분리한다.
- 타입, 스킬, 상태이상 이름은 문자열 직기입보다 `id 참조`를 우선한다.
- 불리언, 세대명, 슬롯값은 표기 규칙을 초반에 고정한다.
- 비어 있는 값을 허용할 컬럼과 반드시 채워야 하는 컬럼을 구분한다.

권장 규칙:

- `monster_id`, `skill_id`, `type_id`는 소문자 스네이크 케이스
- 세대는 `gen1`, `gen2`처럼 고정
- 불리언은 `true/false`
- 슬롯은 `1/2/3/passive`처럼 명시

## 작업 순서 제안

### 단계 1. 현재 README 표를 기준으로 1기 후보를 확정

우선 `Example Modelmon`, `Starter Trio`, 스킬/타입 표를 참고해 `1기 몬스트 명단`부터 잠근다.

### 단계 2. 몬스터 본체 CSV 먼저 작성

가장 먼저 `gen1_monsters.csv`를 만들고, 각 몬스터의 ID와 기본 스탯, 역할을 정리한다.

### 단계 3. 타입/스킬 매핑 CSV 분리

중첩 정보는 본체 CSV 안에 `type1,type2,skill1,skill2...`로 밀어 넣기보다 별도 매핑 파일로 뺀다.

### 단계 4. 이후 런타임 변환 계층 추가

나중에는 CSV를 읽어 게임에서 쓰는 JS 객체나 JSON으로 변환하는 작은 스크립트를 두는 것이 좋다.

예상 흐름:

`CSV -> 검증 스크립트 -> 런타임 JSON 또는 JS 데이터`

## 문서 기준 합의가 필요한 항목

아래는 실제 CSV 작성 전에 한 번 정해야 한다.

- `1기 몬스트`가 정확히 몇 종인지
- 스타터 3종만 먼저 할지, 예시 몬스터 전체를 넣을지
- 타입을 몬스터 본체 컬럼에 둘지, 별도 매핑 테이블로 바로 분리할지
- 스킬도 공용 사전으로 뺄지, 1기 전용으로 단순 시작할지
- 스탯 체계를 4개로 갈지, 더 세분화할지

## 추천 결론

현재 프로젝트 단계에서는 아래 접근이 가장 안전하다.

1. `docs/`에 방향성 문서를 남긴다.
2. `data/` 또는 `docs/data/` 아래에 `gen1_monsters.csv`부터 만든다.
3. 타입과 스킬은 처음부터 별도 매핑 CSV로 분리한다.
4. README 표는 소개용으로만 유지하고, 수치 데이터 기준은 CSV로 옮긴다.
