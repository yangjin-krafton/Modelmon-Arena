# Modelmon Arena 전투 대사 연출 기획서

## 목적

이 문서는 나중에 Modelmon Arena가 실제 대전 기능을 갖추게 되었을 때,
LLM의 가장 큰 장점인 `문자열 생성 능력`을 전투 연출에 적극 활용하기 위한 기획 문서다.

목표는 단순히

- "공격했다"
- "상태 이상에 걸렸다"
- "데미지를 입었다"

수준의 건조한 로그를 출력하는 것이 아니라,
포켓몬식 턴제 전투 구조를 유지하면서도
각 모델몬의 개성, 스킬의 성격, 상태 이상의 분위기를
`대사(dialog)`와 `짧은 서술`로 더 재미있게 전달하는 것이다.

## 핵심 방향

전투 문자열 시스템은 아래 두 축을 함께 사용한다.

1. `사전 제작 문자열 라이브러리`
2. `상황 기반 동적 문자열 생성기`

이 둘을 섞는 이유는 명확하다.

- 라이브러리만 쓰면 안정적이지만 금방 반복적으로 느껴진다.
- LLM 생성만 쓰면 비용, 속도, 품질 편차, 세계관 일관성 문제가 생긴다.

따라서 기본 구조는 다음과 같다.

- 자주 발생하는 장면은 라이브러리에서 빠르게 호출한다.
- 중요한 장면, 희귀 조합, 연승/역전/상성 카운터 같은 특별한 순간은 동적으로 생성한다.
- 생성 결과가 좋으면 다시 라이브러리에 편입해 자산으로 축적한다.

즉, 전투 문자열은 매번 새로 쓰는 것이 아니라
`라이브러리를 성장시키는 생성 파이프라인`으로 운영한다.

## 왜 이 방식이 Modelmon Arena에 맞는가

Modelmon Arena의 모티프는 AI 모델과 AI 능력 그 자체다.
그래서 전투의 재미도 수치 계산만이 아니라,
각 행동이 "어떤 AI적 개성으로 보이느냐"에서 크게 올라간다.

예를 들어 같은 공격이라도

- 생성 계열 모델은 화려하고 과장된 출력 폭주처럼,
- 추론 계열 모델은 차분하지만 정확한 반격처럼,
- 검색 계열 모델은 약점을 포착한 분석 공격처럼,
- 에이전트 계열 모델은 연쇄 지시와 실행 흐름처럼

다르게 느껴져야 한다.

이 차이를 가장 싸고 유연하게 표현하는 수단이 문자열이다.

## 연출 레이어

전투 텍스트는 하나의 긴 문단이 아니라, 여러 레이어로 분리해서 관리한다.

### 1. 시스템 로그 레이어

판정 전달이 목적이다.

- 누가 행동했는가
- 어떤 스킬을 썼는가
- 어떤 결과가 났는가
- 상태 이상이 적용되었는가

예시:

- `GPT Sprout used Prompt Burst.`
- `Claude Shell entered Guard Mode.`
- `Gemini Ember was afflicted with Drift.`

이 레이어는 항상 짧고 명확해야 하며, 스킵 가능해야 한다.

### 2. 대사 레이어

캐릭터성과 감정 전달이 목적이다.

- 사용 직전 한마디
- 피격 반응
- 상태 이상에 걸렸을 때의 혼란/분노/침묵
- 막타 직전 또는 역전 직후의 짧은 대사

예시:

- `"이번 출력은 꽤 자신 있는데."`
- `"응답 흐름이 꼬였어... 잠깐만."`
- `"그 약점, 이미 분석 끝났어."`

### 3. 장면 서술 레이어

화면에 보이는 전투 연출을 짧게 묘사한다.

- 스킬의 시각적 느낌
- 상태 이상이 퍼지는 방식
- 방어막/충전/폭주 같은 장면성

예시:

- `공중에 떠오른 프롬프트 조각이 한 줄로 정렬되며 적을 향해 쏘아졌다.`
- `노이즈가 화면 가장자리부터 번지며 입력 흐름을 흐렸다.`

### 4. 강조 컷신 레이어

항상 나오지 않고, 특정 조건에서만 발동하는 특수 문장이다.

- 첫 킬
- 마지막 1:1 구도
- 약점 상성 대처 성공
- 연속 상태 이상 유발
- 치명타
- 체력 20% 이하 생존 후 반격

이 레이어는 가장 LLM 활용 가치가 높다.

## 문자열 생성 방식

### A. 완전 고정형

완전히 수작업으로 작성한 문자열을 사용한다.

적합한 경우:

- 튜토리얼
- 자주 보는 기본 공격
- UI 핵심 판정 문장
- 번역이 꼭 필요한 공용 문장

장점:

- 빠르다
- 통제가 쉽다
- 테스트가 쉽다

단점:

- 반복 피로가 생긴다

### B. 슬롯 조합형

미리 준비한 조각을 조합해 한 문장을 만든다.

예시 구조:

- 시작 톤
- 행동 동사
- 스킬 키워드
- 결과 문장

예시:

- `[차분한 선언] + [정밀하게 겨눈다] + [추론 파동] + [약점을 찔렀다]`

이 방식은 비용이 거의 없고 변주가 쉽다.
기본 전투의 60~70%는 이 방식으로 커버하는 것이 좋다.

### C. 조건 프롬프트형 LLM 생성

전투 상태를 요약한 작은 입력을 넣고,
1~3문장의 짧은 전투 서술을 생성한다.

적합한 경우:

- 희귀 조합
- 스토리성 있는 매치업
- 보스전
- 결정적인 역전 장면
- 같은 행동이 반복되어 텍스트 피로가 쌓일 때

입력 예시:

- 공격자 정체성
- 수비자 정체성
- 사용 스킬의 성격
- 현재 HP 비율
- 상태 이상 여부
- 이번 장면의 분위기 태그

출력 예시:

- 사용 직전 대사 1줄
- 장면 서술 1줄
- 결과 반응 1줄

### D. 사후 선별형

LLM이 만든 결과를 바로 실시간 사용하지 않고,
내부 검수 또는 자동 필터를 거쳐 라이브러리에 저장한다.

이 방식은 시즌 운영에 특히 유리하다.

- 좋은 문장은 재사용 자산이 된다.
- 이상한 문장은 버리면 된다.
- 세계관 어조를 점점 고정할 수 있다.

## 권장 아키텍처

권장 흐름은 아래와 같다.

1. 전투 엔진이 `이벤트 데이터`를 만든다.
2. 텍스트 연출기가 이벤트 중요도를 판정한다.
3. 중요도가 낮으면 라이브러리/슬롯 조합형 문장을 사용한다.
4. 중요도가 높으면 LLM 생성 후보를 요청한다.
5. 안전성/길이/톤 필터를 통과한 문장만 노출한다.
6. 만족도가 높은 문장은 별도 라이브러리 후보로 저장한다.

즉, LLM은 전투 판정 엔진을 대체하지 않는다.
판정은 엔진이 하고, 표현만 LLM이 맡는다.

## 이벤트 분류

텍스트 연출 우선순위를 이벤트별로 다르게 잡는다.

### 1. 기본 공격

가장 빈도가 높다.

- 기본은 슬롯 조합형
- 가끔만 동적 생성

### 2. 강한 필살기

전투의 기억점이 되기 쉽다.

- 전용 라이브러리 문장 다수 확보
- 연출 등급이 높을 때 LLM 생성 허용

### 3. 상태 이상 부여

Modelmon Arena에서 특히 중요하다.
AI 세계관과 잘 맞기 때문이다.

예:

- Hallucination
- Drift
- Lag
- Overfit
- Silence
- Desync

상태 이상은 단순 효과보다
"무슨 식으로 망가지고 있는가"를 문장으로 보여주는 것이 핵심이다.

예시 방향:

- `Lag`: 반응이 한 박자씩 늦게 돌아온다
- `Hallucination`: 없는 해답을 본 것처럼 엇나간다
- `Drift`: 원래 의도와 다른 문맥으로 미끄러진다
- `Overfit`: 지금 상황보다 익숙한 패턴에만 집착한다

### 4. 방어/카운터/회피

대전에서 손맛이 큰 부분이다.

- 읽기 성공
- 상성 대응
- 예측 방어
- 치명타 방지

이런 장면은 짧아도 인상이 강하다.

### 5. 교체/입장

포켓몬식 전투 리듬에서 매우 중요하다.

- 입장 대사
- 필드 장악감
- 카운터 출전 느낌

### 6. 전투 전환점

가장 비싼 연출을 써도 되는 지점이다.

- 마지막 1체
- 연속 KO
- 역전
- 체력 임계점 돌입
- 특수 상태 중첩

## 상태 이상 묘사 원칙

상태 이상 텍스트는 전투 연출의 핵심 자산이다.
각 상태마다 아래 4종 세트를 확보하는 것을 권장한다.

1. `적용 순간 문장`
2. `턴 시작 문장`
3. `행동 실패/약화 문장`
4. `해제 순간 문장`

예시: `Lag`

- 적용: `신호가 밀리기 시작했다. 응답 속도가 눈에 띄게 처진다.`
- 턴 시작: `명령은 들어갔지만 반응이 늦다.`
- 행동 실패: `타이밍이 어긋나며 기술이 반 박자 늦게 새어 나갔다.`
- 해제: `지연이 걷히며 흐름이 다시 맞아 들어간다.`

예시: `Hallucination`

- 적용: `존재하지 않는 정답을 좇기 시작했다. 판단이 흔들린다.`
- 턴 시작: `시야는 또렷한데 결론만 어긋난다.`
- 행동 실패: `엉뚱한 해석에 끌려가며 공격 축이 비틀렸다.`
- 해제: `허상이 걷히고 실제 신호가 다시 잡힌다.`

## 톤 가이드

전투 텍스트는 아래 기준을 지킨다.

- 1~2문장 중심으로 짧게 쓴다.
- 지나치게 장황한 문학체는 피한다.
- 로그 판정과 감정 대사를 분리한다.
- 모델몬의 계열별 어조 차이는 살린다.
- 유머는 가능하지만 밈 과다 사용은 피한다.
- 플레이어가 여러 턴을 반복해도 피로하지 않아야 한다.

계열별 예시 톤:

- `General`: 안정적, 자신감, 표준적
- `Reasoning`: 차분함, 논리적, 정밀함
- `Generation`: 화려함, 과잉 출력, 감각적
- `Retrieval`: 분석적, 냉정함, 약점 포착
- `Agent`: 지시, 절차, 연쇄 실행
- `Alignment`: 보호, 제어, 안정화
- `Hallucination`: 불안정, 왜곡, 불길함

## 금지 규칙

전투 문자열 생성에는 명확한 금지선이 필요하다.

- 실제 브랜드/기업/정치/사회 이슈를 직접 풍자하지 않는다.
- 플레이 흐름을 끊을 정도로 긴 문장을 출력하지 않는다.
- 승패 판정을 모호하게 만드는 표현을 쓰지 않는다.
- 전투 시스템에 없는 효과를 암시하지 않는다.
- 동일 문장을 너무 자주 반복하지 않는다.
- 캐릭터 어조가 갑자기 바뀌지 않게 한다.

## 데이터 설계 제안

문자열 리소스는 이벤트 단위로 관리한다.

```json
{
  "eventType": "status_apply",
  "statusId": "lag",
  "speaker": "target",
  "tone": "frustrated",
  "priority": "common",
  "tags": ["realtime", "delay", "tempo-loss"],
  "lines": [
    "신호가 밀린다. 박자가 늦어지고 있어.",
    "반응이 따라오지 못한다. 흐름이 끊겼다."
  ]
}
```

추가 메타데이터 예시:

- `modelmonLine`
- `skillId`
- `archetype`
- `hpBand`
- `battlePhase`
- `rareTrigger`
- `canReuse`
- `localizationKey`

## 동적 생성 프롬프트 초안

실시간 생성 시에는 긴 자유 생성보다
짧은 구조화 프롬프트를 권장한다.

입력 예시:

- 공격자 이름: `Gemini Ember`
- 공격자 계열: `Generation / Realtime`
- 대상 이름: `Claude Shell`
- 사용 기술: `Prompt Flare`
- 결과: `critical hit`
- 현재 분위기: `momentum swing`
- 문장 수: `최대 3줄`
- 금지: `판정 왜곡 금지, 과도한 농담 금지`

출력 요구:

- 1줄: 사용 직전 짧은 대사
- 1줄: 장면 묘사
- 1줄: 피격 또는 결과 반응

## 화면 노출 방식 제안

UI에서는 모든 문장을 한꺼번에 길게 보여주지 않는다.

권장 방식:

- 1단: 시스템 로그
- 2단: 대사 말풍선
- 3단: 선택적으로 장면 서술 오버레이

즉, 플레이어는 판정을 놓치지 않으면서도
원할 때 더 풍부한 연출을 즐길 수 있어야 한다.

옵션 예시:

- `Fast`: 시스템 로그 중심
- `Normal`: 로그 + 핵심 대사
- `Cinematic`: 로그 + 대사 + 장면 서술 + 특수 컷신

## 제작 우선순위

처음부터 모든 상황을 LLM으로 처리할 필요는 없다.
아래 순서가 현실적이다.

### Phase 1

- 기본 공격/피격/KO 고정 문장 작성
- 상태 이상 6~10종의 4세트 문장 작성
- 계열별 어조 가이드 확정

### Phase 2

- 슬롯 조합형 문자열 시스템 구현
- 스킬 태그 기반 문장 선택 구현
- 희귀 이벤트 판정 로직 추가

### Phase 3

- 일부 이벤트에 한해 LLM 생성 연결
- 결과 필터링 및 길이 제한 도입
- 좋은 결과를 라이브러리로 재수집하는 운영 루프 추가

## 기대 효과

- 전투가 같은 규칙이어도 매 턴의 인상이 달라진다.
- Modelmon의 AI 개성이 숫자 밖에서도 살아난다.
- 상태 이상과 상성전이 더 기억에 남는다.
- 적은 그래픽 리소스로도 연출 밀도를 높일 수 있다.
- 나중에 PvP, PvE, 보스전 모두에 확장 가능하다.

## 최종 정리

Modelmon Arena의 전투 텍스트는
그냥 로그가 아니라 `게임의 또 다른 전투 자산`으로 취급해야 한다.

가장 좋은 구조는

- 기본은 빠르고 안정적인 문자열 라이브러리,
- 변주는 슬롯 조합,
