# 모델몬 도감 개발 지침

## 목적

이 문서는 앞으로 `모델몬 도감`을 실제 게임 개발의 출발점으로 삼기 위한 기준을 정리한다.

현재 프로젝트의 첫 MVP 목표는 전투 시스템이 아니라,  
`모델몬이 어떤 존재인지 한눈에 이해되는 도감 경험`을 만드는 것이다.

즉, 도감은 단순한 설정 문서가 아니라 다음 개발의 기반 데이터 역할을 해야 한다.

- 캐릭터 정체성 확정
- 아트 자산 정리
- 이름과 세계관 톤 통일
- 이후 전투/스킬/상성 시스템으로 확장 가능한 데이터 기반 마련

## 현재 합의된 방향

### 1. 첫 MVP는 도감 우선

초기 목표는 배틀 구현보다 `모델몬 도감` 완성도에 둔다.

우선 확인해야 할 것은 다음이다.

- 각 모델몬이 시각적으로 구분되는가
- 이름이 기억에 남는가
- AI 모델 개념이 캐릭터에 자연스럽게 대입되는가
- 같은 계열 안에서 진화 흐름이 느껴지는가

### 2. 도감 데이터가 기획의 기준점이다

앞으로 캐릭터 정보는 README 표보다 구조화된 데이터 기준으로 관리한다.

현재 샘플 기준 데이터 위치:

- [modelmon-dex-001-009.csv](/mnt/d/Weeks/Modelmon-Arena/sandbox/modelmon-dex-mvp/data/modelmon-dex-001-009.csv)
- [index.html](/mnt/d/Weeks/Modelmon-Arena/sandbox/modelmon-dex-mvp/index.html)

README는 소개용이고,  
실제 캐릭터 정의는 CSV 같은 데이터 파일이 기준이 된다.

### 3. 모든 사용자 노출 문구는 한국어 우선

도감 화면, 카드명, 라벨, 필터, 설명 문구는 한국어를 기본으로 한다.

영문 표기는 내부 식별자나 보조 표기 수준으로만 유지한다.

## 모델몬 설계 원칙

### 1. 원작 포켓몬 인상 + AI 모델 평가를 혼합한다

모델몬은 단순히 AI 모델 이름을 붙인 패러디가 아니다.

각 캐릭터는 아래 두 층을 함께 가져야 한다.

- 원작 포켓몬에서 오는 실루엣, 성격, 성장 인상
- 실제 AI 모델에 대한 대중적 평가, 장점, 약점, 사용감

예를 들면:

- 빠르고 가벼운 모델은 작은 체급, 민첩한 성향
- 신뢰성과 정렬성이 강한 모델은 방어적이거나 차분한 인상
- 멀티모달 모델은 감각 기관이 확장된 형태
- 고성능 대형 모델은 존재감이 크지만 무겁고 비용이 큰 인상

### 2. 이름은 한국어로 재치 있게 변환한다

모델명은 그대로 쓰지 말고,  
읽는 맛이 있는 한국어식 도감 이름으로 바꾼다.

현재 예시:

- `GPT-3.5` → `지피티삼점오`
- `GPT-4` → `지피티포`
- `GPT-4o` → `지피티포오`
- `Claude Haiku` → `클로드하이쿠`
- `Claude Sonnet` → `클로드소네트`
- `Claude Opus` → `클로드오푸스`
- `Gemini Nano` → `제미니나노`
- `Gemini Pro` → `제미니프로`
- `Gemini Ultra` → `제미니울트라`

원칙:

- 발음이 자연스러워야 한다
- 너무 길지 않아야 한다
- 한국어 UI에 넣었을 때 리듬이 살아야 한다
- 필요하면 이후 더 캐릭터적인 별칭으로 발전시킬 수 있다

### 3. 계열과 진화 흐름이 보여야 한다

도감 번호가 이어질 때는 한 계열 안에서 성장 흐름이 보여야 한다.

현재 기준:

- `001~003`: OpenAI 시리즈
- `004~006`: Claude 시리즈
- `007~009`: Gemini 시리즈

한 계열 안에서는 다음이 자연스럽게 이어져야 한다.

- 외형 체급
- 개념 확장
- 설명 밀도
- 존재감

## 도감 데이터 기준 필드

현재 도감 MVP 기준 필드는 아래를 우선한다.

- `dex_no`: 도감 번호
- `modelmon_id`: 내부 식별자
- `name_ko`: 한국어 이름
- `name_en`: 원본 모델명 또는 영문 표기
- `stage`: 단계
- `core_concept`: 핵심 AI 개념
- `sub_concept`: 보조 AI 개념
- `input_mode`: 입력 방식
- `output_mode`: 출력 방식
- `model_params`: 모델 파라미터
- `file_size_kb`: 실제 파일 크기
- `motif`: 캐릭터 모티프 설명
- `temperament`: 성향
- `flavor_text_ko`: 도감 설명
- `image_path`: 이미지 경로

## 필드 해석 원칙

### 1. 속성 대신 AI 개념을 쓴다

기존 게임식 원소 타입보다,  
`AI 모델이 무엇을 잘하는가`를 직접 드러내는 개념을 우선한다.

예시:

- `대화`
- `추론`
- `생성`
- `코드`
- `정렬`
- `검색`
- `멀티모달`
- `실시간`
- `장문맥`
- `도구사용`
- `인프라`

### 2. 키는 모델 파라미터로 대체한다

도감에서 물리적 키를 보여주기보다,  
모델의 체급감을 드러내는 `모델 파라미터`를 사용한다.

예시:

- `7B`
- `16B`
- `32B`
- `70B`
- `120B`

### 3. 몸무게는 실제 파일 크기로 대체한다

물리적 몸무게 대신,  
현재 샘플 단계에서는 연결된 원본 이미지의 실제 파일 크기를 사용한다.

이 값은 임시이지만 다음 장점이 있다.

- 자산 기준으로 측정 가능하다
- 샘플 제작 단계에서 바로 계산 가능하다
- 도감 항목이 완전히 빈 설정값이 되지 않는다

주의:

장기적으로는 이 값을 `모델 용량감`, `배포 무게`, `연산 비용` 같은  
세계관 값으로 다시 바꿀 수 있다.

### 4. 모티프, 성향, 설명은 혼합 창작 원칙을 따른다

`motif`, `temperament`, `flavor_text_ko`는 아래를 섞어서 만든다.

- 원작 포켓몬의 외형과 분위기
- 해당 AI 모델의 일반적 인상
- 실무 사용자들이 느끼는 장단점

즉, 설명은 단순 요약이 아니라  
`이 모델이 왜 이런 캐릭터가 되었는가`를 보여주는 문장이어야 한다.

## 제거하기로 한 항목

현재 도감 MVP에서는 아래 항목을 쓰지 않는다.

- 분류
- 계열
- 태그
- 전투 스탯
- 희귀도
- 스킬 목록

이 항목들은 나중에 필요해지면 다시 추가하되,  
지금은 도감의 핵심 전달력을 해치는 중복 정보로 본다.

## 실제 개발 시 작업 순서

### 1단계. 번호와 시리즈 고정

먼저 도감 번호 범위와 실제 대응 시리즈를 고정한다.

예:

- `001~003`: OpenAI
- `004~006`: Claude
- `007~009`: Gemini

### 2단계. 이름 확정

한국어 이름을 먼저 확정한다.

이름은 이후 UI, 카드, 설명, 썸네일, 파일명 논의까지 영향을 주므로  
초기에 흔들리지 않게 잡는 편이 좋다.

### 3단계. AI 개념 필드 입력

각 모델몬에 대해 아래를 먼저 채운다.

- 핵심 개념
- 보조 개념
- 입력 방식
- 출력 방식
- 모델 파라미터

### 4단계. 설명문 작성

그 다음 `모티프`, `성향`, `도감 설명`을 작성한다.

이때 중요한 기준:

- 설명은 짧아도 캐릭터성이 살아야 한다
- 단순 홍보 문구처럼 쓰지 않는다
- 장점만 적지 말고 인상적 약점도 함께 드러낸다

### 5단계. 이후 게임 시스템으로 연결

도감이 어느 정도 쌓이면 그 다음에 아래 항목을 분리 설계한다.

- 전투 역할
- 스킬
- 상태이상
- 상성
- 진화 조건

즉, `도감 -> 캐릭터성 -> 시스템화` 순서로 간다.

## 구현 지침

### 1. 파일 직접 실행도 가능해야 한다

샌드박스 실험 화면은 가능하면 `file://`로도 열리게 유지한다.

그래서 현재 도감 샘플은:

- 외부 CSV 파일
- HTML 내부 내장 CSV

를 함께 두는 구조를 사용하고 있다.

### 2. 원본 이미지는 직접 참조한다

샘플 단계에서는 이미지를 중복 복사하지 않는다.

현재 기준 원본 경로:

- `sandbox/originals/001.png ~ 009.png`

### 3. 화면보다 데이터 구조를 먼저 흔들어 본다

도감 MVP 단계에서는 화려한 프론트엔드보다  
데이터 구조가 오래 버티는지가 더 중요하다.

즉, 먼저 검증할 것은:

- 번호 체계
- 이름 체계
- 개념 분류
- 설명 톤
- 시리즈 대응 방식
